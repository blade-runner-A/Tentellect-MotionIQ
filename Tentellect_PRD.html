Context & Strategic Reframe

The original plan assumed hardware-first: build a device, deploy to a factory, collect data, then train models. This PRD inverts that order. The insight is simple — you cannot pitch to robotics companies with a diagram. You pitch with a running system.

This document defines the pipeline required to go from zero to a working, demo-ready skeleton AI system using entirely free public datasets. The device and factory deployment come after you have traction to show.

Strategic Goal

Build a complete skeleton extraction, auto-annotation, and action recognition pipeline on free public industrial data. Use this as the technical proof to unlock access to robotics companies' proprietary datasets and factory deployments — without spending money on hardware or factory negotiations first.

Key Constraint

Two engineers. No dedicated GPU cluster. No factory access yet. Every architectural decision must reflect this reality. Colab Pro ($10/month) and Vast.ai spot instances ($0.30/hr) are the compute budget. Everything else is free.

01

Goal & Success Definition

By end of Week 8, the system must be able to accept any industrial video or image folder as input and return:

  • A COCO-format keypoints JSON with 33-landmark skeleton per detected worker per frame
  • Per-keypoint confidence scores with automatic accept/review/discard classification
  • PPE compliance score per worker (helmet, vest, gloves detected relative to skeleton)
  • A 20-dimension feature vector per worker per 1-second window (spatial + temporal)
  • Action class prediction (fall / bend / walk / reach / idle / danger posture)
  • Risk score ∈ [0.0, 1.0] per worker per window
  • Auto-annotation export in COCO JSON, YOLO TXT, and CSV formats
  • A Label Studio review queue for all frames below the confidence threshold
The Demo Deliverable

The output of Week 8 is a single script: python run_pipeline.py --input /path/to/video that processes any factory video and produces annotated skeletons, risk scores, and a dataset export — in under 5 minutes for a 1-hour video. This is what you show robotics companies.

Data Quality Gates

Every frame entering the pipeline passes through three gates. No frame that fails all three gates enters the training dataset.

Gate Condition Threshold Action on Pass Action on Fail
G1 — Detection At least one person detected with bbox confidence > 0.60 Proceed to G2 Discard frame
G2 — Pose Confidence Mean keypoint confidence across all 17 critical joints > 0.85 → auto-accept
0.50–0.85 → review
< 0.50 → discard		 Auto: write to dataset
Review: Label Studio queue		 Discard
G3 — Keypoint Plausibility Anatomical validity check — limb length ratios within ±30% of biomechanical norms, no self-intersecting skeleton Pass/fail rule-based check Accept with G2 decision Route to review regardless of G2
Agent Task — Quality Gate Implementation

Implement src/annotation/quality_gates.py with a QualityGate class that takes a frame's keypoint dict (COCO format) and returns one of AUTO_ACCEPT, REVIEW, or DISCARD with a reason string. The plausibility check must validate: (1) shoulder width / height ratio in [0.2, 0.8], (2) arm length symmetry within 20%, (3) no keypoint coordinate outside frame bounds, (4) minimum 8 of 17 keypoints visible. Write pytest tests covering edge cases: fully occluded worker, multiple workers, headless detection.

Accepts any video file (MP4, AVI, MKV), image folder, RTSP stream URL, or COCO JSON directory. Normalizes all inputs to a standard frame stream before skeleton extraction.

Frame Extraction Strategy
ModeSourceFPSResolutionUse Case

Preprocessing — CLAHE for Factory Lighting
src/ingestion/preprocess.py — core function import cv2 import numpy as np def preprocess_frame(frame: np.ndarray) -> np.ndarray: """ Apply CLAHE (Contrast Limited Adaptive Histogram Equalization) to compensate for uneven factory floor lighting. Converts to LAB, equalizes L channel only, converts back. Dramatically improves pose estimation under fluorescent/shadow conditions. """ lab = cv2.cvtColor(frame, cv2.COLOR_BGR2LAB) l, a, b = cv2.split(lab) # clipLimit=2.0 prevents over-amplification of noise # tileGridSize=(8,8) means 64 local regions per frame clahe = cv2.createCLAHE(clipLimit=2.0, tileGridSize=(8,8)) l_eq = clahe.apply(l) merged = cv2.merge([l_eq, a, b]) return cv2.cvtColor(merged, cv2.COLOR_LAB2BGR) def normalize_resolution(frame: np.ndarray, target: tuple = (640, 480)) -> np.ndarray: """Letterbox resize — no distortion, pads with grey.""" h, w = frame.shape[:2] scale = min(target[0]/w, target[1]/h) new_w, new_h = int(w*scale), int(h*scale) resized = cv2.resize(frame, (new_w, new_h)) canvas = np.full((target[1], target[0], 3), 114, dtype=np.uint8) x0 = (target[0] - new_w) // 2 y0 = (target[1] - new_h) // 2 canvas[y0:y0+new_h, x0:x0+new_w] = resized return canvas

Skeleton Extraction

Two-model approach: YOLOv8-Pose for multi-person detection and 17 COCO keypoints, MediaPipe Pose for 33-landmark detail and world 3D coordinates. ByteTrack handles identity persistence across frames.

Model Selection Logic
ConditionPrimary ModelReason
1 person in frame MediaPipe Pose Faster, 33 landmarks, world 3D coords, more accurate for single person
2+ persons in frame YOLOv8-Pose Multi-person native, handles occlusion between workers better
Low confidence (<0.7) Both + ensemble Average predictions where both models agree, flag disagreement
Video sequence (not image) YOLOv8-Pose + ByteTrack Tracking requires detection model — MediaPipe is single-frame only
Skeleton Output Schema
Output JSON schema per detection — stored in SQLite + COCO JSON { "frame_idx": 1420, "timestamp_ms": 47333, "session_id": "sh17_batch_001", "track_id": "worker_003", "bbox": [124, 88, 312, 450], # [x1, y1, x2, y2] "detection_confidence": 0.94, "keypoints_17": [ # COCO 17 keypoints {"name": "nose", "x": 218, "y": 102, "conf": 0.97}, {"name": "left_shoulder", "x": 167, "y": 189, "conf": 0.92}, # ... 15 more keypoints ], "keypoints_33": [/* MediaPipe 33 landmarks */], "world_3d": [ # MediaPipe world coordinates (metres) {"name": "nose", "x": 0.02, "y": -0.44, "z": -0.08}, # ... 32 more world landmarks ], "mean_confidence": 0.88, "quality_gate": "AUTO_ACCEPT", "ppe": { "helmet": 0.94, "vest": 0.87, "gloves": 0.21, "glasses": 0.05 } }

08

IMU Fusion Layer

When IMU data is available (OpenMarcie dataset has synchronized IMU, future field device will have it), this layer fuses accelerometer readings with skeleton keypoints to produce occlusion-robust skeleton enrichment and physics-based auto-labels (fall, impact, rapid movement).

When IMU data is not available (COCO, SH17, iSafetyBench processing), this layer is skipped and IMU features are zeroed in the feature vector with an imu_available: false flag.

IMU Auto-Labels (Free Annotation from Physics)
LabelIMU ConditionConfidenceAction
fall_event |accel| > 2.5g for 50ms then < 0.5g for 200ms High Auto-label sequence. No human review needed.
impact_detected |accel| > 3.0g single spike High Flag 2s clip around spike for human review
rapid_movement SMA (Signal Magnitude Area) > 2.0 over 1s window Medium Flag as high-activity period in metadata
static_posture SMA < 0.3 for > 30s High Auto-label idle / standing still
bending_forward pitch > 35° sustained for > 3s Medium Flag for ergonomic risk annotation
Agent Task — IMU Fusion Module

Implement src/imu/fusion.py with two classes: (1) ComplementaryFilter that takes ax/ay/az at variable sample rates and outputs pitch/roll in degrees, using alpha=0.98. (2) IMUAutoLabeler that takes a buffer of IMU readings (list of dicts with ax/ay/az/timestamp_ms) and returns a list of AutoLabel dataclasses with fields: label_type, confidence, start_ms, end_ms, metadata. Must handle the case where IMU and video are on different sample rates — interpolate IMU to match video frame timestamps using numpy interp. Include a test_imu_fusion.py that generates synthetic fall signature data and verifies the fall_event label fires correctly.

09

Auto-annotation & Review System

The annotation system has two paths: auto-accepted frames go directly to dataset storage; review-flagged frames are pushed to a local Label Studio instance for human correction. The goal is less than 10 minutes of human annotation per hour of processed video.

Label Studio Integration
  1. 1
    Run Label Studio locally
    label-studio start --port 8080 — runs entirely on your machine, no cloud account needed
  2. 2
    Pipeline pushes review tasks via SDK
    When G2 returns REVIEW, the pipeline calls label_studio_sdk to create a task with the frame image + auto-skeleton overlay as the pre-annotation. Reviewer sees what the model got and just corrects it.
  3. 3
    Reviewer corrects keypoints (target: 30s per frame)
    Pre-annotated skeletons mean reviewers drag existing points, not place them from scratch. 30s/frame is achievable when >80% of keypoints are already correct.
  4. 4
    Completed reviews pulled back via SDK and merged into dataset
    Reviewed annotations replace auto-predictions in SQLite and are flagged human_verified: true. These count double in training sample weighting.
Agent Task — Annotation Module

Implement src/annotation/annotator.py with a AnnotationRouter class. It takes a skeleton dict and quality gate result and routes to either: (1) DatasetWriter — writes to SQLite and appends to COCO JSON in-memory, (2) LabelStudioPusher — creates a Label Studio task with the frame image encoded as base64 and the skeleton as pre-annotation in Label Studio's keypoint format. The LabelStudio project must use the KeypointLabels template with all 17 COCO keypoints defined. Write a scripts/pull_reviews.py that fetches completed tasks from Label Studio and merges them back into the SQLite database, marking records as human_verified=True.

Known Risks & Mitigations

RiskLikelihoodImpactMitigation
NTU RGB+D / Human3.6M request takes > 2 weeks Medium Medium Action classifier can be trained on iSafetyBench alone for Week 6–7 while waiting. Submit both requests in Week 1.
Auto-accept rate below 70% on SH17 Medium Low Lower G2 threshold temporarily to 0.75 for SH17 batch — these are studio-quality images so model should perform well. Investigate which keypoints are failing.
Vast.ai GPU spot instance preempted mid-training Medium Low All training scripts must checkpoint every 5 epochs. Use --resume flag on restart. Always use on-demand instances for the final training run.
ST-GCN action classifier underperforms on industrial video Medium Medium Fall back to a simpler rule-based action classifier using keypoint angles if ST-GCN doesn't hit targets. Rule-based is not elegant but it works and is explainable.
OpenMarcie IMU format incompatible with fusion layer Low Low Write a format adapter in src/imu/adapters/openmarcie.py to normalize to internal IMU dict format before fusion.
No factory video to demo with publicly High Medium Use iSafetyBench clips for demo (already processed and have ground truth). Alternatively, use publicly available YouTube factory footage — search "manufacturing floor time-lapse" for clean material.
Hard Stop

Do not begin fine-tuning (Week 6) until the dataset has at least 5,000 auto-accepted frames from SH17 with quality gate passing. Training on low-confidence auto-annotations produces a model that is worse than the pretrained baseline — this is the most common mistake in fast-moving annotation pipelines.
